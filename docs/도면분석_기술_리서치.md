## 1. 한국 아파트 도면 특성

### 면적 체계 (개발자 필수 이해)

| 구분 | 의미 | 서비스 적용 |
|------|------|-----------|
| **전용면적** | 현관문 안쪽 실거주 공간 (발코니 제외) | AI 분석 핵심 단위 |
| 서비스면적 | 발코니 (무상 제공, 확장 가능) | 확장 여부에 따라 실면적 변동 |
| 공급면적 | 전용 + 주거공용 (계단, 복도 등) | 분양 표기 기준 |

> **실무 주의:** "84㎡ 아파트" = 전용면적 기준, 공급면적으로는 34평형.
> 사용자 입력 시 어떤 면적 기준인지 UI에 명시 필요.

### 평형별 일반 구조 특징

| 평형 | 전용면적 | 방 구성 | 베이 | 주요 특징 |
|------|---------|--------|------|---------|
| 25평형 | 59㎡ | 방 2개+거실+욕실 1~2 | 2~3베이 | 신혼·1~2인, LDK 통합 多 |
| **33평형 (국민평형)** | **84㎡** | **방 3개+거실+욕실 2** | **3~4베이** | **가장 표준화, 샘플 도면 1순위** |
| 45평형 | 114~132㎡ | 방 4개+거실+욕실 2~3 | 4베이 | 드레스룸·팬트리 별도 |

> **베이(bay):** 전면 발코니 기준 기둥~기둥 사이 구획. 3베이 = 거실+방+방이 전면 배치.

### 도면에서 읽을 수 있는 정보 vs 없는 정보

**✅ 읽을 수 있는 것:**
- 방 개수 및 각 실 명칭 (MBR/BR/욕실/주방 등 텍스트)
- 문 위치 (반원 호선 기호)
- 창문 위치 (외벽 끊김 표시)
- 내력벽/비내력벽 구분 (선 굵기 차이)
- 발코니 위치 및 범위
- 공간별 면적 수치 (텍스트 병기 시)
- 배관 위치 (욕실·주방 심벌)

**❌ 읽을 수 없는/어려운 것:**
- 실제 층고 (평면도에 미표기)
- 마감재 종류 (별도 마감표 필요)
- 천장 높이, 보 위치
- 전기 콘센트·스위치 위치 (별도 전기도면)
- 실사용 순면적 (벽체 두께 제외 후 계산 필요)

### 표준 기호 및 약어

| 기호/약어 | 의미 |
|---------|------|
| MBR (Master Bedroom) | 안방 |
| BR (Bedroom) | 침실 |
| LDK | 거실+식당+주방 통합 |
| B/T (Bath/Toilet) | 욕실 |
| WIC (Walk-In Closet) | 드레스룸 |
| **반원 기호** | **문 (여닫이 방향 표시)** |
| **굵은 선** | **내력벽 (철거 불가)** |
| **얇은 선** | **비내력벽 (철거 가능)** |
| 외벽 끊김 | 창문 위치 |
| 점선 박스 | 붙박이 가구/가전 위치 |
| N 화살표 | 북향 기준선 (채광 판단) |

### 도면 이미지 품질 이슈

| 입력 유형 | 주요 문제 | 대응 방법 |
|---------|---------|---------|
| 스마트폰 촬영본 | 원근 왜곡, 조명 불균일, 해상도 부족 | 300dpi 이상, 수직 촬영 안내 |
| PDF 캡처 (저해상도) | JPEG 압축 아티팩트, 텍스트 뭉개짐 | 150dpi 이상 PNG 추출 |
| 분양 카탈로그 스캔 | 컬러 노이즈, 인쇄 망점 | 그레이스케일 변환 후 처리 |
| 벡터 PDF | 선 굵기 정보 소실 | SVG 또는 고해상도 PNG 변환 |

> **Claude Vision 기준:** 이미지 최대 8000×8000px. 1568px 이상은 자동 다운스케일.
> **도면 텍스트가 포함된 경우 단변 1000px 이상 유지 필수.**

## 2. Claude Vision 도면 분석 능력 벤치마크

### AECV-bench 실측 성능 (2025~2026)

| 모델 | 전체 평균 | 침실 인식 | 화장실 인식 | 문 인식 | 창문 인식 |
|------|---------|---------|-----------|--------|---------|
| **Gemini 2.5 Pro** | **41%** | ~78% | ~74% | ~26% | ~14% |
| GPT-5 | 37% | 높음 | 높음 | ~12% | 낮음 |
| **Claude 3.7 Sonnet** | **35%** | 양호 | 양호 | 낮음 | 낮음 |
| Llama 4 Maverick | 11% | - | - | - | - |

**핵심 인사이트:**
- 침실·화장실: 텍스트 레이블 기반 → OCR 수준 인식 가능 (70~78%)
- 문·창문: 건축 기호(arc, 끊김선) 기반 → **극도로 낮은 인식률 (12~26%)**
- 공간 영역(폴리곤) 추출: ~15%로 매우 낮음

### 흔히 실패하는 케이스

1. **문/창문 기호 오인식**: 호선(arc)·외벽 끊김을 기하학적 패턴으로 인식 실패
2. **내력벽/비내력벽 구분 실패**: 이미지 다운스케일 후 선 굵기 차이 소실
3. **면적 수치 오독**: 기울어진 텍스트, 소수점 숫자 (3.25㎡ 등)
4. **유사 공간 혼동**: 창고·다용도실을 침실로 분류
5. **발코니 경계 혼동**: 실내와 발코니 경계 구분 실패

## 5. SVG 2D 배치도 자동 생성

### Python 라이브러리 스택

| 라이브러리 | 역할 |
|---------|------|
| `shapely` | 2D 기하 연산, 방 폴리곤, 가구 충돌 감지 |
| `svgwrite` | SVG 파일 생성 (순수 Python) |
| `ezdxf` | CAD DXF 파싱·생성, SVG 변환 |
| `cairosvg` | SVG → PNG 렌더링 |

### 핵심 구현 코드

```python
import svgwrite

def rooms_to_svg(rooms_json: list, output_path: str):
    dwg = svgwrite.Drawing(output_path, profile='full')

    ROOM_COLORS = {
        'MBR': '#FFE4C4',
        'BR': '#E6E6FA',
        'LivingRoom': '#F0FFF0',
        'Kitchen': '#FFF8DC',
        'Bathroom': '#E0F0FF',
        'Balcony': '#F5F5F5',
    }

    for room in rooms_json:
        bbox = room['position']['bbox_normalized']
        color = ROOM_COLORS.get(room['type'], '#FFFFFF')

        x1, y1, x2, y2 = [v * 500 for v in bbox]
        w, h = x2 - x1, y2 - y1

        dwg.add(dwg.rect(
            insert=(x1, y1), size=(w, h),
            fill=color, stroke='#333', stroke_width=1.5
        ))
        dwg.add(dwg.text(
            room.get('korean_label', room['type']),
            insert=(x1 + w/2, y1 + h/2),
            text_anchor='middle', font_size='10px'
        ))

    dwg.save()
```

## 7. Vision R&D 구현 로드맵

```
Phase 1 — 도면 입력 처리
├── PIL/OpenCV로 원근 왜곡 보정, 해상도 정규화
├── 품질 검사: 단변 1000px 미만 시 경고
└── Claude Vision API 호출 (base64 또는 Files API)

Phase 2 — MVP 정보 추출 (현재 목표: 방 이름·방 개수 80%+)
├── 시스템 프롬프트: 한국 도면 기호 규칙 명시
├── Few-shot: LH 공공데이터 기반 평형별 2~3개 예시
├── 프롬프트 분해: 방 이름 추출 → 방 개수 확인 → 주요 방 우선순위
└── JSON Structured Output 강제

Phase 3 — 방별 이미지 생성용 컨텍스트 구성
├── 거실·주방·안방·작은방 우선순위 선정
├── 선택 톤과 방 용도를 Imagen 프롬프트로 변환
└── 욕실·발코니·현관·다용도실은 MVP 렌더링 제외

Phase 4 — SVG 배치도 생성 (Phase 2 이후)
├── shapely로 방 폴리곤 계산
├── svgwrite로 2D 배치도 렌더링
└── 가구 배치: Rule-based CSP 또는 LLM 추천

Phase 5 — 정확도 보완
└── 문·창문·내력벽까지 다룰 때 사용자 확인 요청 UI 검토
```

### Evaluation Set 구성 (KDT 발표 핵심 근거)

```
샘플 도면: 10~15장 (25평·33평·45평 각 3~5장)
정답 JSON: 수동 작성 (방 개수·용도·면적 수동 레이블링)
평가 지표:
  - 방 개수 정확도 (Exact Match)
  - 방 용도 분류 F1 Score
  - 면적 추정 MAPE (Mean Absolute Percentage Error)
  - 문/창문 위치 정확도 (IoU)
```
