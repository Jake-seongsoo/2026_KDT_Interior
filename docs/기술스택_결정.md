```
     ├─→ [GCP Cloud Storage]  도면 임시 저장 (30일 TTL 자동 삭제)

```

```
└── 환경변수:  Cloud Run 환경변수 직접 설정 + Vercel 환경변수
```

### CORS 설정 (Cloud Run ↔ Vercel)

FastAPI에서 Vercel 도메인을 허용해야 합니다.

```python
# main.py
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
  CORSMiddleware,
  allow_origins=[
    'https://<your-app>.vercel.app',
    'http://localhost:3000',   # 로컬 개발
  ],
  allow_methods=['*'],
  allow_headers=['*'],
)
```

### Cold Start 대응 (Cloud Run)

Cloud Run은 트래픽 없을 때 컨테이너를 종료 → 첫 요청 1~3초 지연 발생.
KDT 발표 당일 데모 전 워밍업 요청 1회 필요.

```bash
# 발표 직전 워밍업
curl https://<cloud-run-url>/health
```

최소 인스턴스 1개 유지 옵션 (미미한 비용, 발표용으로만 설정):
```bash
gcloud run services update <service> --min-instances=1
```

### Supabase JWT → Cloud Run 인증 흐름

```
사용자 로그인 (Google OAuth)
    └── Supabase Auth → JWT 발급
         └── Next.js: Authorization: Bearer <jwt> 헤더에 포함
              └── Cloud Run FastAPI: Supabase JWT 검증
                   └── supabase-py verify_token() 또는 python-jose 직접 검증
```

## 안 비교 — 안 C 선택 근거

3가지 안을 검토한 결과 안 C (GCP + Supabase 하이브리드)를 채택했습니다.

| 항목 | 안 A (GCP 풀) | 안 B (Supabase 풀) | **안 C 채택** |
|------|:---:|:---:|:---:|
| 월 인프라 비용 | ~$10 (Cloud SQL) | ~$0 | **~$0** |
| 초기 설정 시간 | 2~3일 | 0.5일 | **1일** |
| 도면 30일 자동 삭제 | ✅ 자동 | ⚠️ 수동 코드 필요 | **✅ 자동 (Cloud Storage Lifecycle)** |
| Google OAuth 설정 | NextAuth (1일) | Supabase (30분) | **Supabase (30분)** |
| Imagen 연동 | ✅ GCP 동일 프로젝트 | ⚠️ Railway → Vertex AI 별도 설정 | **✅ GCP 동일 프로젝트** |

**안 A 탈락 이유:** Cloud SQL 최소 $7~10/월 고정 비용, KDT 포트폴리오 단계에서 불필요
**안 B 탈락 이유:** Supabase Storage 자동 만료 미지원 → 개인정보보호법 준수를 위한 삭제 스케줄러 직접 구현 필요
