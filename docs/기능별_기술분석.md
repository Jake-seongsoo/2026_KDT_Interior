## fc1 — 도면 이미지 업로드

### 구현 기술

```
[클라이언트] JPG/PNG 5MB 이내 → multipart/form-data
[서버] FastAPI → 이미지 검증 → Cloud Storage 임시 저장
[전처리] PIL/OpenCV: 해상도 정규화, 1000px 이상 단변 유지
[출력] base64 인코딩 → Claude Vision API 전달
```

**필수 전처리 항목:**
- 단변 1000px 미만 → 업로드 전 경고 (Claude Vision 자동 다운스케일 기준 1568px)
- 촬영 도면: 원근 왜곡 보정 (OpenCV perspectiveTransform)
- 스마트폰 HEIC → PNG 자동 변환

### 현재 기술적 제약

| 제약 | 원인 | 대응 |
|------|------|------|
| 손그림·저화질 스캔에서 인식률 급락 | 노이즈·압축 아티팩트 | 150dpi 이상 PNG 추출 가이드 제공 |
| 벡터 PDF에서 선 굵기 소실 | 래스터화 시 정보 손실 | SVG 또는 고해상도 PNG 변환 후 업로드 권장 |
| 이미지 최대 8000×8000px | Claude Vision 사양 | 초과 이미지 자동 리사이즈 |

### 유사 서비스 구현 방식

**Planner 5D** — 도면 업로드 후 1~2분 처리, 완료 시 이메일 알림 발송. 자체 CV 파이프라인. 손그림 스케치도 지원하나 정확도 경고 제공.

**Houzz Pro (AutoMate AI)** — 2D 도면을 즉시 3D로 변환. 벽·문·창문 핵심 건축 요소를 자동 보존(preserve). 내부 기술 스택 비공개.

**시사점:** 처리 중 로딩 UI와 "도면 품질 기준 안내 화면"이 이탈률에 직접 영향. Planner 5D는 이메일 알림을 쓸 정도로 처리 시간이 길고, 우리 목표인 30~60초와 방향이 다름. 빠른 처리를 위해 Vision API에 직접 연결하는 방식이 타당.

## fc2 — Claude Vision 방 구성 분석

### 구현 기술

```
[입력] 전처리된 도면 이미지 (base64)
[AI] Claude Vision API (claude-3-7-sonnet) + MVP용 3단계 프롬프트
  - Step 1: 도면에 표시된 한글 방 이름 추출
  - Step 2: 방 개수와 주요 방 우선순위 산정
  - Step 3: 거실/주방/안방/작은방 등 이미지 생성 대상 후보 분류
[출력] Structured Output (JSON Schema 강제)
[UI] 사용자 보정 없음. 분석 실패·저신뢰도는 재업로드 안내 또는 낮은 확신도 표시
```

**Few-shot 전략:**
- AI Hub 데이터셋(ID 71465) 또는 LH 공공데이터에서 평형별 예시 도면 2~3장 첨부
- 동적 선택: 입력 도면 평형과 유사한 예시 선택 (25평/33평/45평 각각 준비)

### 현재 기술적 제약 (AECV-bench 실측, 2025~2026)

| 항목 | Claude 3.7 | Gemini 2.5 Pro | 비고 |
|------|-----------|----------------|------|
| 전체 평균 | 35% | 41% | 기본 프롬프트 기준 |
| 침실·욕실 | 70~78% | 70~78% | 텍스트 레이블 기반, 비교적 양호 |
| **문(Door)** | **~26%** | **~26%** | 반원 기호 기하학 패턴 인식 실패 |
| **창문(Window)** | **~14%** | **~18%** | 외벽 끊김 단서에만 의존 |
| 공간 폴리곤 추출 | ~15% | - | 경계 좌표 추출 매우 어려움 |

**흔히 실패하는 케이스:**
1. 발코니·창고를 침실로 분류
2. 기울어진 텍스트 숫자 오독 (3.25㎡ 등)
3. 내력벽/비내력벽 이미지 다운스케일 후 선 굵기 소실
4. 실내와 발코니 경계 구분 실패

**MVP 개발상 핵심 결론:**
- 사용자 보정 UI를 넣지 않기 때문에 입력 범위를 아파트 분양도면·네이버 부동산 캡처로 제한한다.
- MVP의 정확도 목표는 문·창문·내력벽이 아니라 **방 이름과 방 개수**다.
- 문·창문 기반 정밀 배치와 SVG 탑뷰는 Phase 2 이후로 분리한다.

### 유사 서비스 구현 방식

**Planner 5D** — 범용 VLM 아닌 자체 전문 CV 모델. 처리 파이프라인:
1. 전용 CV 모델 → 문·창문·벽체 바운딩박스 추출
2. 비관련 영역 마스킹
3. LLM → 마스킹 이미지에서 텍스트·의미 추출
4. 두 결과 병합 → JSON 구조화

→ **Brain.co 사례**: 이 하이브리드 방식으로 단순 VLM 60% → 97% 정확도 달성

**FloorplanVLM (arXiv 2025)** — 도면 벡터화를 이미지 조건부 시퀀스 모델링으로 접근. 아직 상용 서비스 아닌 연구 단계.

**ZURU Tech (AWS Bedrock)** — Few-shot 프롬프팅으로 109% 정확도 향상 사례. 우리 서비스의 v1→v3 R&D 경로의 근거.

**시사점:** MVP는 순수 Claude Vision + Few-shot으로 시작. Phase 2에서 전용 CV 모델(YOLO 계열) 병용으로 정확도 점프 가능. AI Hub 데이터셋이 이 전환의 핵심 자원.

## fc3 — 도면 기반 톤 6개 추천

### 구현 기술

```
[입력] 도면 분석 JSON + 방 구성 요약 (+ 선택: 레퍼런스 이미지 1장)
[트렌드 수집] Claude Web Search Tool
  - 쿼리 예: "2026 재팬디 인테리어 아파트 트렌드"
  - 쿼리 예: "2026 quiet luxury 한국 아파트"
[톤 6개 생성] Claude LLM 단일 호출
  - 각 톤: 톤명 + 카테고리 + 배색 + 한 줄 설명 + 도면 기반 추천 이유
  - 다양성 규칙: 고급/자연/미니멀/컬러/트렌디/실용 계열이 서로 겹치지 않도록 강제
  - 레퍼런스 모드: 6개 전부 레퍼런스 시그니처(primary_hex·materials·style_tokens·mood) 시드로 변주
[사용자 선택] 6개 톤 중 1개 선택
[렌더링] 선택 톤으로 주요 방 최대 4장 생성 (방별 1장)
  - 레퍼런스 없음: Imagen 4 (imagen-4.0-generate-001) 텍스트 프롬프트 생성
  - 레퍼런스 있음: Imagen 3 Capability (imagen-3.0-capability-001) StyleReferenceImage conditioning
```

**레퍼런스 이미지 선택 흐름 (추가 옵션):**
```
[도면 업로드] + [레퍼런스 이미지 업로드 (선택)]
        │                      │
        │              Claude Vision → 톤 시그니처 추출
        │              { primary_hex, secondary_hex, accent_hex,
        │                materials[], style_tokens[], lighting, mood }
        │                      │
        └──────────────────────┘
                   │
           Claude 톤 6개 생성 (시그니처 기반 변주)
                   │
             사용자 톤 1개 선택
                   │
        Imagen 3 Capability + StyleReferenceImage [1]
              (레퍼런스 색감·재질·조명 conditioning)
```

**병렬 구조 (속도 최적화):**
```
Vision rooms ──→ Web Search 트렌드 키워드
        │                 │
        └────→ Claude 톤 후보 6개 생성
                          │
                    사용자 톤 선택
                          │
          ┌───────────────┼───────────────┐
          ▼               ▼               ▼
      거실 이미지       주방 이미지       안방/작은방 이미지
      (Imagen)          (Imagen)          (Imagen)
```

### 현재 기술적 제약

| 제약 | 원인 | 대응 |
|------|------|------|
| 6개 톤 간 차별성 부족 | LLM이 유사한 뉴트럴 스타일로 수렴하는 경향 | 고급/자연/미니멀/컬러/트렌디/실용 카테고리 강제 |
| Web Search 비용 | 트렌드 검색마다 ~$0.01 발생 | 트렌드 결과 캐싱 (24시간 TTL) |
| Imagen 생성 시간 | 주요 방 최대 4장 병렬 기준 60초 이내 목표 | 프로그레시브 로딩 UI (먼저 완성된 방부터 표시) |
| 실제 공간 치수 미반영 | Imagen이 도면 JSON을 직접 이해 못함 | 프롬프트에 "방 너비 X, 천장 Y" 텍스트 명시 |
| 레퍼런스 모드 렌더링 품질 ↓ | Imagen 3 Capability < Imagen 4 일반 품질 | 레퍼런스 없는 사용자는 기존 Imagen 4 경로 유지. "스타일 일치 우선 모드" UX 안내 |
| 레퍼런스 저작권 리스크 | SNS·블로그 이미지 무단 업로드 가능 | 업로드 UI 동의 문구 + private 버킷 + 30일 자동 삭제 |

### 유사 서비스 구현 방식

**Havenly AI (가장 체계화된 사례)** — 2.4백만 건 자체 디자인 데이터 + 고객 반응 데이터(좋아요/수정 이력)로 LLM 파인튜닝. 텍스처·색상 이론·공간 흐름·라이프스타일 이해 보유 표방. 구체적 모델 아키텍처 비공개.

**FlairGPT (arXiv 2025)** — LLM에게 오브젝트 목록과 배치 제약 조건을 구조화된 방식으로 생성하도록 유도 → 레이아웃 그래프로 변환. "LLM 단독으로는 완성 레이아웃 불가, 반드시 제약 조건 레이어 필요"라는 것이 핵심 발견.

**Interior AI** — 30+ 스타일 프리셋. Web Search 연동 없이 사전 정의된 스타일 카테고리 사용. 실시간 트렌드 반영은 확인 불가.

**시사점:** Web Search 실시간 트렌드를 스타일 생성에 연동하는 서비스는 조사 범위 내 없음. 이 자체가 차별점. 단, 트렌드 캐싱과 비용 관리 필수.

## fc4 — 실구매 상품 실시간 추천

### 구현 기술

```
[입력] 선택 톤 + 방 이름 + 예산 범위
[API] 네이버쇼핑 검색 API (무료 25,000회/일)
  GET https://openapi.naver.com/v1/search/shop
  params: query="호텔라이크 거실 소파", display=10, sort=sim
[상품 파싱] 상품명 HTML 태그 제거 + 가격 범위 정리
[출력] 방별 추천 상품 목록 (이름·가격·이미지·구매링크·판매처)
[면책] "가격·재고는 검색 시점 기준, 클릭 시 판매처 확인"
```

**상품 매칭 전략:**
- 방별 검색 쿼리 템플릿: `{선택 톤} {방 이름} {가구 유형}` (예: "호텔라이크 거실 소파")
- MVP 추천 수량: 방별 3~5개 상품
- 가격 필터: lprice 기준 예산 범위 내 필터링 (API 파라미터로 불가 → 클라이언트 필터)
- 재고 확인: API가 재고 반환 안 함 → 링크 클릭 시 판매처에서 확인

### 현재 기술적 제약

| 제약 | 원인 | 대응 |
|------|------|------|
| **재고 상태 알 수 없음** | Naver API 미제공 | "클릭 시 판매처에서 확인" 안내 (면책 필수) |
| AI 생성 가구 ↔ 실제 상품 시각 갭 | 렌더링 가구가 "이상적 버전" | 상품 이미지를 렌더링과 나란히 표시해 갭 인식 |
| 검색어 의존성 | 쿼리 품질이 결과 품질 결정 | 스타일별 쿼리 템플릿 사전 작성 + A/B 테스트 |
| 할인 상품 식별 불가 | API 할인 정보 미제공 | lprice/hprice 범위로 대체 표시 |

### 유사 서비스 구현 방식

**Havenly AI (가장 발전된 방식):**
1. AI가 "이상적 가구"를 먼저 가상 생성 (알고리즘으로 생성)
2. 50만 건 이상의 실제 카탈로그에서 시각적으로 가장 유사한 상품 매칭
3. → 매칭 알고리즘 세부 기술 비공개, 임베딩 유사도 검색으로 추정

**오늘의집** — 자사 커머스 + 3D 시뮬레이션(아키스케치 파트너십) 통합. 실제 제품 3D 모델을 가상 배치 후 구매 연동. AI 개인화 추천으로 재구매율 증가 효과 자체 발표.

**IKEA Kreativ** — IKEA 자사 카탈로그만 연동. 사용자 방 사진 업로드 → AR로 자사 가구 가상 배치.

**네이버쇼핑 API + 인테리어 AI 연동 사례** — 조사 범위 내 없음. 우리 서비스가 국내 선행 사례가 될 가능성.

**시사점:** Havenly의 "가상 가구 생성 → 실제 상품 매칭" 구조는 정확하나 구축 비용이 큼. MVP는 네이버 검색 쿼리 방식으로 시작, Phase 2에서 임베딩 유사도 매칭 고도화 검토.

## fc5 — 톤 재추천 + 정밀화

### 구현 기술

```
[피드백 입력] 👍/👎 버튼 + 거절 이유 태그 선택
  태그 예: "너무 어두워서", "가격이 높아서", "모던하지 않아서"
[컨텍스트 누적] 세션 내 대화 이력 + 거절 이유 리스트 → Claude 컨텍스트
[재추천] 기존 거절 이유를 Negative Constraint로 추가해 톤 6개를 재생성
[정밀화 파라미터] 예산 범위 슬라이더 + 가족 형태 + 취향 키워드 입력
```

**Negative Constraint 프롬프트 예:**
```
이전에 다음 이유로 거절된 스타일을 피하세요:
- 너무 어두운 색상 (대신 밝고 따뜻한 톤 우선)
- 가격 300만원 이상 소파 (대신 150만원 이하 우선)
이 제약을 지키면서 새로운 톤 6개를 생성하세요.
```

### 현재 기술적 제약

| 제약 | 원인 | 대응 |
|------|------|------|
| Positive 선호 학습 어려움 | "싫어하는 것"은 명확, "좋아하는 것"은 모호 | 긍정 선택 시 "어떤 점이 좋은가?" 추가 질문 |
| 세션 간 선호 미유지 | 기본 무상태(stateless) | Phase 3 fc9(이력서)로 해결 |
| API 비용 폭증 | 재추천마다 방별 Imagen 호출 가능 | MVP처럼 톤 선택 전에는 이미지 생성하지 않음 |
| 재추천 다양성 부족 | 컨텍스트가 길어질수록 LLM이 보수적으로 수렴 | 주기적 컨텍스트 리셋 옵션 제공 |

### 유사 서비스 구현 방식

**Interior AI** — 생성 모드 4가지 분기 (스타일 변경/공간 변형/빈 공간 채우기/처음부터 생성). 버튼 하나로 다른 경로의 재추천.

**Havenly AI** — AI 결과 불만족 시 인간 디자이너로 seamless 전환. AI + 인간 하이브리드 피드백 루프.

**SIGIR 2025 Agentic Feedback Loop 연구:**
- 추천 에이전트 + 사용자 에이전트 상호작용 구조
- 단일 추천 에이전트 대비 평균 11.52% 개선
- 단일 사용자 에이전트 대비 21.12% 개선

**Remodel AI Pro** — 월정액 구독으로 무제한 생성. 비용 문제를 구독 모델로 해결.

**시사점:** Phase 2에서 "텍스트 먼저, 이미지 나중" 흐름으로 비용 절감하면서 피드백 루프 빠르게 돌릴 수 있음.

## fc7 — 이유 있는 큐레이션 (추천 근거 제시)

### 구현 기술

```
[추천 근거 생성] Claude LLM
  - 입력: 도면 분석 JSON + 선택 톤 + 방 이름
  - 출력: 방별 "왜 이 톤이 맞는가" 2~3줄 근거
  - 예: "거실 3.6m 폭에 2인 소파가 딱 맞고, 창문이 남향이라 원목 마루가 따뜻하게 보임"

[거절 이유 태그] 사전 정의 카테고리
  - 색상 계열: "너무 어두운 톤", "차갑게 느껴짐"
  - 예산: "예산 초과", "가성비 우선"
  - 스타일: "너무 모던함", "더 따뜻한 느낌 원함"
```

**근거 생성의 핵심 원칙:** "이 공간의 구체적 수치"를 근거에 반드시 포함. "좋아 보여서"는 근거가 아님.

### 현재 기술적 제약

| 제약 | 원인 | 대응 |
|------|------|------|
| Post-hoc 근거 문제 | LLM이 결정 후 이유를 꾸미는 경향 | 근거 생성을 스타일 선택 전에 수행 (추론 우선) |
| 근거와 실제 알고리즘 불일치 | LLM이 도면 수치를 정확히 연산 못함 | 도면 수치를 프롬프트에 명시적으로 전달 |
| 거절 이유 → 개선 연결 부족 | 거절 이유가 다음 추천에 미반영 | fc5 피드백 루프와 명시적 연결 |

### 유사 서비스 구현 방식

**Havenly AI** — LLM 파인튜닝 + 50만 건 카탈로그 RAG 조합으로 "왜 이 소파인가" 텍스트 자동 생성.

**ReimagineHome** — 스타일 설명 텍스트 제공. 구체적 근거 수준보다는 스타일 설명 수준.

**국내 서비스 현황** — LLM 기반 추천 근거 텍스트를 제공하는 국내 인테리어 서비스 없음.

**시사점:** 차세대 표준은 LLM + RAG (상품 속성 DB 검색 → "이 제품이 당신 공간에 어울리는 이유" 텍스트 생성). MVP에서는 도면 수치 기반 간결한 근거 3줄로 시작.

## fc8 — 공정별 예산 사전 추정

### 구현 기술

```
[입력] 평수 + 지역 + 시공 범위 (도배/바닥/욕실 등 체크박스)
[단가 계산] Python 상수 (UNIT_COSTS, REGIONAL_MULTIPLIER)
  → 상세 코드: 공정별단가_트렌드_리서치.md
[출력] 공정별 최저~최고 범위 + 우선순위 조언
  예: "이 예산으로 도배+강마루는 가능, 욕실은 부분 교체만 현실적"
[면책] ±30% 오차 안내 + "배관·전기 현장 상황에 따라 추가 발생 가능"
```

**예산 구간별 조언 로직:**
- 300만원 이하: 단일 공정만 권장 (도배 또는 장판 교체)
- 500만~1,500만원: 2~3개 공정 조합 가능
- 1,500만~3,000만원: 20~25평 기준 올수리 수준
- 3,000만원 이상: 전체 리모델링 + 고급 자재 가능

### 현재 기술적 제약

| 제약 | 원인 | 대응 |
|------|------|------|
| 숨겨진 공사비 반영 불가 | 배관·전기 상태는 현장에서만 파악 가능 | "현장 상황에 따라 추가 발생" 명시 |
| 실시간 단가 연동 불가 | 공종별 단가 API 없음 | 하드코딩 상수 + 분기별 수동 업데이트 |
| 이미지 기반 물량 자동 산출 불가 | 도면에서 정밀 면적 추출 어려움 | 사용자 직접 평수 입력 |

### 유사 서비스 구현 방식

**Decor8 AI Remodel Cost Estimator:**
- 공사 유형 20+ 선택 + ZIP코드 입력 → 2분 이내 결과
- 지역별 인건비·자재비 자동 보정 (샌프란시스코 vs 중서부 농촌 55% 차이 반영)
- 예산/중급/프리미엄 3개 옵션으로 인건비·자재비·허가비·예비비 세부 항목 출력

**집닥 (국내)** — AI 자동 추정 아닌 전담 플래너 1:1 견적 분석 모델. 순수 AI 공정별 자동 추정 없음.

**국내 AI 자동 예산 추정** — 국내 서비스 중 AI 자동화 공종별 예산 추정은 없음.

**시사점:** 한국 시장 특화 지역 보정계수(서울/수도권/지방광역시/기타)를 이미 확보했으므로 Decor8 AI보다 국내 정확도에서 유리한 시작점. ZIP코드 수준의 세밀한 지역 구분은 Phase 2 고도화 항목.

## fc10 — 내 도면으로 번역하기 (SNS to My Room)

### 구현 기술

```
[입력] 도면 이미지 + SNS 영감 이미지 1~3장
[Vision A] Claude Vision — 도면 분석
  + 벽 분류 (외벽/코어인접벽/세대간벽/추정내력벽/칸막이벽)
[Vision B] Claude Vision — 영감 이미지 분석
  스타일·색상·가구·조명·재질 추출
[호환성 매핑] Claude LLM
  입력 조건 vs 영감 요소 → 가능/불가/조건부 분류 + 대안 생성
[출력] 3-zone 화면
  LEFT: 원본 영감 + 추출 스타일 태그
  CENTER: Gemini Imagen 렌더링 (내 도면 버전)
  RIGHT: ✅가능 요소·❌불가 요소+대안·⚠️확장 가능 영역·🛒쇼핑 링크
```

**벽 분류 로직:**

| 벽 유형 | 식별 단서 | 확장 가능 | 신뢰도 |
|---------|---------|---------|--------|
| 외벽 | 세대 외곽 가장 바깥 굵은 선 | ❌ 절대 불가 | 100% |
| 코어 인접벽 | 엘리베이터·계단실 옆 굵은 벽 | ❌ 절대 불가 | 100% |
| 세대간벽 | 옆집 공유 외곽 벽 | ❌ 절대 불가 | 100% |
| 추정 내력벽 | 굵은 이중선 + 해칭 패턴 | ⚠️ 불가 가능성 높음 | 70~85% |
| 칸막이벽 | 얇은 단일선, 해칭 없음 | ✅ 철거 가능 | 70~85% |

**필수 면책 (모든 출력 강제 포함):**
> "벽 분류는 도면 두께·패턴 기반 AI 추정입니다. 실제 시공 전 관리사무소·구조 기술사 확인이 필수입니다."

### 현재 기술적 제약

| 제약 | 원인 | 대응 |
|------|------|------|
| 분양 카탈로그는 구조 도면이 아님 | 약식 도면 — 두께 표기 부정확 | 한계 명시 + "관리사무소 구조 도면 확인" 권장 |
| Vision A·B 2회 호출 비용 | 도면 + 영감 이미지 각 Vision 호출 | 비동기 병렬 처리로 시간 최소화 |
| 영감 이미지 저작권 | 오늘의집·인스타 이미지 원저작권 | 약관에 "저작권 침해 책임은 사용자" 명시 |
| 내력벽 오판 시 법적 책임 | 잘못된 철거로 안전사고 | 면책 문구 + 신뢰도 % 강제 표시 + 확정 케이스(100%)와 추정 케이스(70~85%) 시각 구분 |

### 유사 서비스 구현 방식

**ReimagineHome (가장 근접한 사례):**
- 영감 이미지 업로드 → 실제 방 이미지와 병합
- 낮은 천장·비대칭 창문 등 실제 제약 내에서 작동
- 기본값: 레이아웃 유지 + 마감재·조명·오브젝트만 변경 (구조 변경 없음)
- ControlNet 계열 기술 사용 추정

**Interior AI — ControlNet 조합:**
- Inpainting ControlNet: 선택 영역 채우기
- Canny ControlNet: 기존 선·엣지 강화
- Depth ControlNet: 3D 원근감 유지

**"가능/불가 분류" 기능** — 조사 범위 내 어떤 서비스도 영감 이미지 적용 가능/불가를 명시적으로 분류하는 기능 없음. **미개척 영역.**

**벽 구조 분석 상용 서비스** — 소비자 대상으로 제공하는 서비스 없음. 학술 연구 수준에서만 존재 (ScienceDirect 2024). 기업들이 안전사고 법적 책임으로 기피하는 영역.

**시사점:** fc10은 기술적으로 가장 도전적이나, 시장 공백이 가장 크고 P05(SNS 기대치 괴리) 해결의 핵심. Phase 3으로 분류한 것은 fc2 Vision 정확도가 80%+ 달성 후 벽 분류 신뢰도를 확보해야 하기 때문.

## 구현 우선순위 결정 근거

### 높은 우선순위 (기술 성숙 + 국내 공백)

1. **fc3 톤 6개 추천** — 도면 기반으로 매번 다른 톤을 추천하는 흐름이 MVP 선택 경험의 핵심.
2. **fc4 방별 상품 추천** — 네이버쇼핑 API 연동 국내 인테리어 AI 없음. 구현 2~3일.
3. **fc7 추천 근거** — 국내 LLM 기반 추천 근거 제시 없음. fc3과 동시 구현 가능.

### 중간 우선순위 (기술 도전 있으나 MVP 핵심)

4. **fc2 Vision 방 구성 분석** — 사용자 보정 UI 없이 동작해야 하므로 입력 도면 제한과 실패 처리 UX가 중요.
5. **선택 톤 기반 방별 이미지 생성** — 주요 방 최대 4장으로 비용·대기 시간을 제한.

### 낮은 우선순위 (기술 미성숙 또는 Phase 3)

6. **fc10 SNS 번역** — 벽 분류 정확도 확보 후 구현. 미개척이지만 기반 기술(fc2) 먼저 필요.
7. **fc9 이력서** — 회원 기반 확보 후 의미 있음.
