# Moodie 배포 · 운영 가이드

이 문서는 **최초 배포 절차**와 **배포 이후 재배포·롤백·운영** 절차를 단계별로 정리한다.

- 백엔드(FastAPI) → **GCP Cloud Run** (GitHub Actions 자동 배포)
- 프론트엔드(Next.js) → **Vercel** (Git 연동 자동 배포)
- DB → **Supabase** (마이그레이션 수동 적용)

> 배포 순서 의존성: **백엔드 배포 → Cloud Run URL 확보 → 프론트 환경변수 주입 → 프론트 배포 → 프론트 도메인을 백엔드 CORS에 추가**. 이 순환을 한 번에 끝내야 정상 동작한다.

---

## 0. 사전 준비 (최초 1회)

### 0-1. gcloud CLI 설치 & 로그인
Windows에서 [Google Cloud SDK 설치](https://cloud.google.com/sdk/docs/install) 후, 프롬프트에 직접 입력:

```bash
! gcloud auth login
! gcloud config set project gen-lang-client-0635611562
```

> `!` 접두사로 실행하면 인터랙티브 로그인 결과가 이 세션에 들어온다.

### 0-2. GCP API 활성화
```bash
gcloud services enable run.googleapis.com artifactregistry.googleapis.com cloudbuild.googleapis.com
```

### 0-3. 배포용 서비스 계정 권한 확인
GitHub Actions가 사용할 계정(`vertex-express@...`)에 아래 역할이 필요하다.

```bash
PROJECT=gen-lang-client-0635611562
SA=vertex-express@${PROJECT}.iam.gserviceaccount.com

# Cloud Run 배포 / 이미지 push / 빌드 / 런타임 SA 사용 권한
for role in roles/run.admin roles/artifactregistry.writer roles/cloudbuild.builds.editor roles/iam.serviceAccountUser roles/storage.admin roles/aiplatform.user; do
  gcloud projects add-iam-policy-binding $PROJECT --member="serviceAccount:${SA}" --role="$role"
done
```

### 0-4. Signed URL 서명 권한 부여 ⚠️ (중요)
Cloud Run 런타임은 개인키가 없어 GCS Signed URL 생성이 **기본적으로 실패**한다(도면 썸네일·렌더 이미지가 안 뜸). 런타임 SA가 **자기 자신을 대신해 서명(signBlob)** 할 수 있도록 권한을 준다.

```bash
gcloud iam service-accounts add-iam-policy-binding $SA \
  --member="serviceAccount:${SA}" \
  --role="roles/iam.serviceAccountTokenCreator"
```

> 이 권한이 있으면 `google-cloud-storage`가 IAM SignBlob API로 Signed URL을 서명한다. 부여 후에도 안 되면 `RISK 메모`(맨 아래) 참조.

## 1. GitHub Secrets 등록 (최초 1회)

저장소 **Settings → Secrets and variables → Actions → New repository secret** 에서 아래를 등록한다. 값은 로컬 `.env` / `backend/service-account.json`에서 복사.

| Secret 이름 | 값 출처 |
|------------|---------|
| `GCP_SA_KEY` | `backend/service-account.json` **파일 전체 내용** |
| `GCP_PROJECT_ID` | `.env`의 `GCP_PROJECT_ID` |
| `GCS_BUCKET_NAME` | `.env`의 `GCS_BUCKET_NAME` |
| `GCS_RENDER_BUCKET_NAME` | `.env`의 `GCS_RENDER_BUCKET_NAME` |
| `ANTHROPIC_API_KEY` | `.env`의 `ANTHROPIC_API_KEY` |
| `SUPABASE_URL` | `.env`의 `SUPABASE_URL` |
| `SUPABASE_ANON_KEY` | `.env`의 `SUPABASE_ANON_KEY` |
| `SUPABASE_SERVICE_ROLE_KEY` | `.env`의 `SUPABASE_SERVICE_ROLE_KEY` |
| `SUPABASE_JWT_SECRET` | `.env`의 `SUPABASE_JWT_SECRET` |
| `CORS_ORIGINS` | 프론트 도메인 확정 후 입력 (1-4 단계에서 갱신) |

> 처음엔 `CORS_ORIGINS`에 `http://localhost:3000` 만 넣고, 프론트 배포 후 Vercel 도메인을 추가한다.

## 2. 백엔드 최초 배포

### 2-A. (권장) 수동 1회 배포로 검증
자동화 디버깅 난이도를 낮추기 위해 **먼저 손으로 한 번** 띄운다.

```bash
gcloud run deploy interior-api \
  --source backend \
  --region asia-northeast3 \
  --allow-unauthenticated \
  --service-account vertex-express@gen-lang-client-0635611562.iam.gserviceaccount.com \
  --memory 1Gi --cpu 1 --timeout 300 --min-instances 0 --max-instances 4 \
  --set-env-vars "ENVIRONMENT=production,GOOGLE_APPLICATION_CREDENTIALS=,GCP_REGION=asia-northeast3,CLAUDE_MODEL=claude-sonnet-4-6,SIGNED_URL_TTL_MINUTES=15,CACHE_TTL_HOURS=24,ENABLE_MULTI_FURNITURE_RECO=true,ENABLE_VISION_RERANK=true" \
  --update-env-vars "^@^CORS_ORIGINS=http://localhost:3000@GCP_PROJECT_ID=<값>@GCS_BUCKET_NAME=<값>@GCS_RENDER_BUCKET_NAME=<값>@ANTHROPIC_API_KEY=<값>@SUPABASE_URL=<값>@SUPABASE_ANON_KEY=<값>@SUPABASE_SERVICE_ROLE_KEY=<값>@SUPABASE_JWT_SECRET=<값>"
```

> `^@^` 는 구분자를 콤마 대신 `@`로 바꾸는 gcloud 문법 — `CORS_ORIGINS` 값에 콤마가 들어가도 안전하다.

배포가 끝나면 출력된 **Service URL**을 메모한다 (예: `https://interior-api-xxxxxxxxxx.asia-northeast3.run.app`).

### 2-B. 헬스체크
```bash
curl https://<cloud-run-url>/health
# 기대: {"status":"ok","environment":"production"}
```

## 3. 프론트엔드 배포 (Vercel)

1. [Vercel](https://vercel.com) → **Add New → Project** → GitHub 저장소 `2026_KDT_Interior` import
2. **Root Directory = `frontend`** 로 지정 (중요 — 모노레포 구조)
3. **Environment Variables** 등록:
   | 이름 | 값 |
   |------|-----|
   | `NEXT_PUBLIC_API_URL` | 2-A에서 얻은 Cloud Run URL |
   | `NEXT_PUBLIC_SUPABASE_URL` | `frontend/.env.local` 값 |
   | `NEXT_PUBLIC_SUPABASE_ANON_KEY` | `frontend/.env.local` 값 |
   | `NEXT_PUBLIC_ENVIRONMENT` | `production` |
4. **Deploy** → 배포 완료 후 Vercel 도메인 확보 (예: `https://moodie.vercel.app`)

---

## 4. 백엔드 ↔ 프론트 연결 마무리 (CORS)

프론트 도메인을 백엔드 `CORS_ORIGINS`에 추가해야 브라우저 호출이 통과한다.

```bash
gcloud run services update interior-api --region asia-northeast3 \
  --update-env-vars "^@^CORS_ORIGINS=http://localhost:3000,https://moodie.vercel.app"
```

그리고 **GitHub Secret `CORS_ORIGINS`** 도 같은 값으로 갱신한다(다음 자동 배포 때 유지되도록).

> ⚠️ 이 단계를 빼먹으면 프론트에서 모든 API가 CORS 차단된다 — 첫 배포 실패의 가장 흔한 원인.

## 5. DB(Supabase) 마이그레이션 점검

`.env`의 `SUPABASE_URL`이 가리키는 실 DB에 `supabase/migrations/0001~0005` 가 모두 적용됐는지 확인한다.

1. Supabase 대시보드 → **SQL Editor**
2. 테이블/컬럼 존재 확인 (`analysis_sessions`, `rooms`, `recommendation_results.refinement_params` 등)
3. 누락분이 있으면 `0001` → `0005` **순서대로** 붙여넣어 실행

## 6. GitHub Actions 자동 배포 활성화 & 검증

1단계의 Secrets가 모두 등록됐다면, `.github/workflows/deploy-backend.yml` 이 **`master` 브랜치에 `backend/**` 변경이 푸시될 때 자동 실행**된다.

### 검증
```bash
# backend/에 사소한 변경(주석 등) 후
git add . && git commit -m "chore: 배포 자동화 검증"
git push main master      # remote 이름이 'origin'이 아니라 'main'임에 유의
```
→ GitHub **Actions 탭**에서 그린 빌드 확인 → Cloud Run에 새 리비전 생성 확인.

> 수동 실행도 가능: Actions 탭 → "Deploy Backend to Cloud Run" → **Run workflow**.

## 7. 재배포 운영 워크플로우 (일상)

배포 이후 **코드 수정 → 안전 반영**하는 표준 절차.

### 7-1. 일반 코드 변경
```
1. feature/기능명  또는  fix/버그명  브랜치 생성
2. 로컬 테스트:  cd backend && pytest        (백엔드)
                cd frontend && npm test     (프론트)
3. master로 PR → 머지
4. 자동 배포:
   - backend/ 변경 → GitHub Actions가 Cloud Run 재배포
   - frontend/ 변경 → Vercel이 자동 재배포
```

### 7-2. DB 스키마 변경 시
- 새 마이그레이션 파일 `supabase/migrations/00XX_설명.sql` 추가
- **코드 배포보다 DB를 먼저 적용** (하위호환 우선)
- 컬럼 추가는 가급적 `nullable` 또는 기본값 지정 → 구버전 코드와 공존 가능하게
- 적용: Supabase SQL Editor에서 실행 (5장 참조)

### 7-3. 환경변수 변경 시 — 갱신 위치 (두 곳 동기화 주의)
| 변수 | 갱신 위치 |
|------|----------|
| 백엔드 런타임 값 (API 키 등) | ① GitHub Secret ② Cloud Run(즉시 반영하려면 `gcloud run services update`) |
| `CORS_ORIGINS` | ① GitHub Secret ② Cloud Run — **양쪽 모두** |
| 프론트 `NEXT_PUBLIC_*` | Vercel 대시보드 환경변수 (변경 후 재배포 필요) |

> GitHub Secret만 바꾸고 재배포 안 하면 Cloud Run엔 반영 안 됨. 반대로 `gcloud run services update`만 하면 다음 자동 배포 때 Secret 값으로 덮어써짐 → **항상 두 곳을 함께** 갱신.

## 8. 롤백 절차

### 8-1. 백엔드 (Cloud Run)
```bash
# 리비전 목록 확인
gcloud run revisions list --service interior-api --region asia-northeast3

# 직전 정상 리비전으로 트래픽 100% 전환
gcloud run services update-traffic interior-api --region asia-northeast3 \
  --to-revisions interior-api-00007-abc=100
```

### 8-2. 프론트 (Vercel)
대시보드 → **Deployments** → 직전 정상 배포 → **⋯ → Promote to Production**

### 8-3. DB
마이그레이션은 자동 롤백이 없다. 되돌림용 SQL을 직접 작성·실행해야 하므로, 파괴적 변경(컬럼 삭제·타입 변경)은 신중히.

## 9. 운영 점검 · 비용

- **헬스체크**: `curl https://<cloud-run-url>/health`
- **로그**: GCP 콘솔 → Cloud Run → `interior-api` → 로그 탭 (또는 `gcloud run services logs read interior-api --region asia-northeast3`)
- **비용 주의**: Claude(분석)·Imagen(렌더링)은 **호출당 과금**. `/analyze` `/render` 호출량이 곧 비용.
- **Cold Start 대응**: 평소 `--min-instances=0`(비용 0). 발표·시연 전 30분에:
  ```bash
  gcloud run services update interior-api --region asia-northeast3 --min-instances 1
  # 워밍업
  curl https://<cloud-run-url>/health
  # 시연 후 원복
  gcloud run services update interior-api --region asia-northeast3 --min-instances 0
  ```

## RISK 메모

- **Signed URL 실패**: 0-4 권한 부여로도 해결 안 되면, ① 런타임 SA 환경변수에 `GOOGLE_APPLICATION_CREDENTIALS`로 Secret Manager에 올린 키 파일을 마운트하거나, ② `storage_service.py`의 `generate_signed_url`에 `service_account_email`+`access_token` 방식(IAM SignBlob)을 명시한다.
- **시크릿 노출**: 현재 워크플로우는 런타임 값을 `env_vars`로 주입한다(Cloud Run 콘솔에서 보임). 보안을 강화하려면 **Secret Manager + `--set-secrets`** 로 전환 — 운영 안정화 후 권장.
- **인증 방식**: GCP 인증은 SA 키(JSON)를 GitHub Secret으로 쓴다. 장기적으로 **Workload Identity Federation**(키 없는 인증)으로 전환하면 키 유출 위험이 사라진다.
